# Repository File Index

This document provides extensive context and detailed descriptions for the primary top-level folders within the repository, acting as a comprehensive reference for their specific use cases and roles in the project's architecture.

## `.worktrees/`
This folder contains active Git worktrees, which are essentially linked checkouts of the repository allowing for simultaneous work on multiple branches without constantly switching contexts. It is particularly useful for managing parallel streams of development, such as investigating complex bugs or implementing isolated features in the Poker GPU environment. For example, specific worktrees are dedicated to fixing heuristic ace rank decoding, resolving preflop all-in scenarios, and handling side-pot showdowns. Each worktree acts as a completely independent working directory with its own isolated state, preventing conflicts and messy stashes in the main checkout. This setup enables developers to quickly pivot between fixing a critical bug in one branch and testing a new feature in another without any friction. It also facilitates easier side-by-side comparisons of different implementations or debugging of cross-branch regressions. The presence of this folder indicates a sophisticated Git workflow aimed at maximizing productivity and minimizing context-switching overhead. By leveraging worktrees, the project maintains a clean and organized approach to concurrent feature development.

## `agents/`
The `agents` directory is the core repository for all Reinforcement Learning agent implementations used throughout the project. It is structurally divided into subdirectories categorizing the agents by their underlying learning algorithms, such as `MonteCarlo` and `TemperalDifference`. Inside these folders, one can find the specific logic governing how these agents interact with their environments, process observations, and update their internal policies or value functions. The implementations here range from foundational algorithms like Q-learning and SARSA to potentially more advanced or customized variants suited for specific games. By isolating the agent logic from the environments, the codebase promotes modularity, allowing the same agent to be easily tested across different simulations like Poker or Blackjack. This directory also serves as a testing ground for experimenting with novel exploration strategies, reward discounting methods, and neural network integrations. Ultimately, it encapsulates the "brain" of the project, housing the intelligent systems that learn to master the provided games. The clean separation of agent types makes it straightforward for new contributors to understand and extend the project's learning capabilities.

## `benchmarking/`
This directory is dedicated to evaluating and quantifying the performance of both the environments and the reinforcement learning agents. It contains specialized scripts and tools designed to stress-test the systems, measure execution times, and track resource utilization (like CPU and GPU usage). A significant focus appears to be on the `Poker` environment, given its complexity and the computationally intensive nature of its state transitions and reward calculations. The benchmarking tools here are crucial for identifying bottlenecks, particularly when optimizing the environment for massively parallel GPU execution. By running standardized tests from this folder, developers can definitively prove whether a newly implemented heuristic or tensor operation has improved the overall throughput. Additionally, these scripts help in comparing the learning efficiency of different agent configurations under controlled conditions. The data generated by these benchmarks directly informs the architectural decisions and refactoring efforts tracked elsewhere in the repository. In essence, it is the project's primary mechanism for ensuring that performance improvements are data-driven and objectively measurable.

## `config/`
The `config` folder centralizes all the hyperparameter definitions and execution settings used by the various environments and agents. It heavily utilizes YAML files (such as `blackjack.yaml`, `pokerGPU.yaml`, and `qlearning.yaml`) to provide a human-readable and easily modifiable format for tweaking the system's behavior. These configuration files define crucial parameters like learning rates, discount factors, exploration thresholds (epsilon), batch sizes, and neural network dimensions. By abstracting these values away from the raw Python code, the project allows for rapid experimentation and hyperparameter tuning without requiring code modifications or recompilation. This separation also ensures that experiments are highly reproducible, as a specific training run can be perfectly recreated simply by loading its corresponding configuration file. Furthermore, the environment-specific configs (like `Particle2d.yaml` or `tfe.yaml`) dictate the rules, grid sizes, reward structures, and other defining characteristics of the simulations. This design pattern greatly enhances the flexibility of the codebase, enabling researchers to swiftly iterate on different experimental setups. Overall, it serves as the master control panel for steering the project's reinforcement learning pipelines.
