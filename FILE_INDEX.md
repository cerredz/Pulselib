# PulseLib File Index

This file is a root-level map of the repository. It is meant to be a compact architecture artifact you can attach to future prompts so agents know where code should go before they start changing anything.

## Root Files

### `.gitignore`
- What it contains: local ignore rules for virtualenv artifacts, generated data, `memory/`, `.claude/`, and the poker evaluator data file.
- Why it matters: it separates durable source code from local workflow artifacts and experiment output.

### `README.md`
- What it contains: currently empty.
- Why it matters: this is the obvious future landing page for project setup, but it is not currently the source of architectural truth.

## Root Folders

### `.claude/`
- What goes here: planning docs, audit reports, reasoning notes, ticket history, and other agent workflow artifacts.
- Importance: this is process memory for AI-assisted work. It is not runtime code, but it explains why recent poker changes happened and preserves investigation context that complements `memory/`.

### `.pytest_cache/`
- What goes here: pytest cache metadata generated by local test runs.
- Importance: disposable tooling output only. It should not be treated as part of the codebase design.

### `agents/`
- What goes here: reinforcement-learning agent implementations grouped by algorithm family, mainly `MonteCarlo/`, `TemperalDifference/`, and `Other/`.
- Importance: this is the policy layer of the repo. `scripts/` uses these agents to train against `environments/`, while `utils/` provides shared agent helpers and defaults.

### `config/`
- What goes here: YAML configuration files for each training path, including poker, poker GPU, blackjack, 2048/TFE, and Monte Carlo experiments.
- Importance: this is the configuration contract between `scripts/`, `environments/`, and `models/`. If you want to understand how a run is wired without reading the full trainer, start here.

### `environments/`
- What goes here: Gymnasium environments for the library's RL tasks, including `2048`, `blackjack`, `Particle2D`, `Poker`, `Tetris`, and `Wordle`, plus environment registration in `__init__.py`.
- Importance: this is the simulation layer and the center of the repo. Agents learn by interacting with these environments, scripts launch them, tests validate them, and configs describe how they should be instantiated.

### `memory/`
- What goes here: long-lived project memory such as milestone notes, tickets, paper summaries, and reusable skill notes.
- Importance: this is the durable harness layer. It captures why the repo evolved the way it did, especially around poker correctness and performance, and is useful context for future agent runs.

### `models/`
- What goes here: reusable neural-network architectures, currently centered on the 2048/TFE models in `tfe.py`.
- Importance: this is the shared model-definition layer. Scripts and agents import from here instead of defining every network inline.

### `results/`
- What goes here: generated experiment artifacts such as learning-curve plots, saved score histories, and poker benchmark YAML runs.
- Importance: this is the output surface for training. `scripts/`, `utils.plotting`, and `utils.benchmarking` write here, so it is where you inspect whether a run actually produced something useful.

### `rl/`
- What goes here: handwritten reinforcement-learning notes and design intuition, including theory notes and poker-specific thinking.
- Importance: this is informal design context. It is not executable code, but it explains the intended learning and reward ideas that shaped the implementations in `agents/`, `environments/`, and `scripts/`.

### `scripts/`
- What goes here: runnable training and watch entrypoints for each environment family, such as `scripts/Poker/trainGPU.py`, `scripts/Poker/train.py`, `scripts/blackjack/train.py`, and `scripts/TFE/train.py`.
- Importance: this is the execution layer. These files compose `config/`, `agents/`, `environments/`, `models/`, and `utils/` into actual training jobs.

### `tests/`
- What goes here: pytest coverage for output services and poker behavior, including round progression, showdown logic, heuristic agents, and trainer-loop behavior.
- Importance: this is the main verification layer. It protects the active poker stack and the shared output utilities from silent regressions.

### `tmp_pr_poker_train_gpu_inner_loop/`
- What goes here: a temporary snapshot of the repository focused on poker GPU inner-loop work.
- Importance: this appears to be an isolated working copy rather than the live source of truth. It is useful for comparison or archival context, but changes should normally go to the main repo tree unless you are intentionally working inside the snapshot.

### `tools/`
- What goes here: local tooling assets, currently including a bundled GitHub CLI and a temporary skill checkout under `tmp_skills/`.
- Importance: this supports the surrounding development harness rather than the runtime library itself. It matters for automation and ticket workflows, not for environment or agent logic.

### `trained_models/`
- What goes here: intended home for persisted trained model checkpoints.
- Importance: this is currently empty, but conceptually it is the durable model-artifact counterpart to `results/`. `results/` holds run outputs; `trained_models/` is where finalized reusable weights would logically live.

### `utils/`
- What goes here: shared infrastructure such as config loading, plotting, benchmarking, replay buffers, device helpers, optimizer helpers, and other reusable training utilities.
- Importance: this is the glue layer across the codebase. Most trainers depend on it to avoid duplicating common concerns like output paths, benchmark summaries, and replay storage.

### `__pycache__/`
- What goes here: Python bytecode cache files.
- Importance: generated interpreter output only. It should be ignored when reasoning about architecture.

## How The Main Source Folders Relate

- `config/` defines run parameters.
- `scripts/` reads `config/` and orchestrates training.
- `scripts/` calls into `environments/`, `agents/`, `models/`, and `utils/`.
- `environments/` defines the tasks and transition logic.
- `agents/` and `models/` define how policies choose actions and learn from transitions.
- `utils/` handles shared operational concerns like plotting, benchmarking, replay buffers, devices, and path resolution.
- `tests/` validates the behavior of the active source tree.
- `results/` and `trained_models/` store outputs produced by that source tree.
- `memory/` and `.claude/` store the project and agent context that explains why the source tree looks the way it does.

## Practical Reading Order

1. `config/`
2. `scripts/`
3. `environments/`
4. `agents/`
5. `utils/`
6. `tests/`
7. `memory/` and `.claude/` for change context

## Current Architectural Center Of Gravity

The repo is broad, but the active center of gravity is the poker stack:

- runtime logic in `environments/Poker/`
- entrypoints in `scripts/Poker/`
- configs in `config/poker.yaml` and `config/pokerGPU.yaml`
- regression coverage in `tests/poker/`
- supporting design and history in `environments/Poker/design.md`, `memory/`, and `.claude/`

That means most high-value changes should assume poker is the most actively maintained path unless the task clearly targets another environment.
